## ⚠️ Global Rules (Apply to ALL Phases)

These rules apply to every single file written in this project. Breaking any of them will cause problems.

1. **Never use `import.meta.env` directly** — always go through `ENV` from `@config/env.js`
2. **Always use import aliases** — `@config/`, `@utils/`, `@components/` — never relative paths like `../../`
3. **All user-visible strings in i18n files** — never hardcode Arabic or English text in JS/HTML
4. **All constants in `constants.js`** — no magic numbers or repeated string literals in code
5. **All Firebase reads go through the services layer** — never call Firestore directly from views or components
6. **All service reads use `cache.js`** — cache wrapper must be applied to protect the 50K reads/day free limit
7. **No `console.log`** — use `Logger` from `@utils/logger.js` (debug in dev, silent in prod)
8. **No writes to Firestore production in V0.1** — admin panel connects to Emulator only (`VITE_USE_EMULATOR=true`)
9. **RTL + LTR must both work** — `dir="rtl"` for Arabic, `dir="ltr"` for English. Test every UI change in both
10. **One file = one responsibility** — if a file is doing two things, split it

### Task 6.1 — `src/js/core/state.js`

**What it is:** A simple reactive state store. Any module can read state, set state, and subscribe to changes. No framework needed.

**Subtasks:**

- [ ] Create `src/js/core/state.js`
- [ ] Implement `getState(key)` — returns current value
- [ ] Implement `setState(key, value)` — only notifies if value actually changed
- [ ] Implement `subscribe(key, callback)` — returns an unsubscribe function
- [ ] Implement `getAllState()` — returns full state snapshot
- [ ] Set initial state: `{ language: 'ar', theme: 'default', isLoading: false, currentRoute: '/', currentUser: null }`

**Coding rules:**

- State keys are fixed — do not allow arbitrary keys from callers. Define them as an object at the top
- Subscribers must be cleaned up — `subscribe()` must return a function that removes the listener
- `setState` must be a no-op if the value is identical to the current value (strict equality)
- Do not use localStorage in this file — `storage.js` handles persistence separately

**Validation:** Import in browser console — `import { getState, setState, subscribe } from '@js/core/state.js'` — subscribe to a key, change it, verify the callback fires.

## Phase 7 — Router System

**What this phase does:** Enables navigation between pages without a full page reload. This is the SPA core.

**Dependencies:** Phase 6 complete (uses `state.js`, `Logger`)

---

### Task 7.1 — `src/js/core/router.js`

**What it is:** A hash-based SPA router. URL format: `/#/university/cu-2024`.

**Subtasks:**

- [ ] Create `src/js/core/router.js`
- [ ] `route(path, handler)` — registers a route pattern
- [ ] `navigate(path, data?)` — changes `window.location.hash`
- [ ] `getCurrentRoute()` — returns current hash path
- [ ] `matchRoute(path)` — extracts params from patterns like `/university/:id`
- [ ] `handleRouteChange()` — internal: matches current path, calls handler, calls `setState('currentRoute', path)`
- [ ] `initRouter()` — attaches `hashchange` listener, calls `handleRouteChange()` immediately
- [ ] `goBack()` — calls `window.history.back()`

**Coding rules:**

- Use `Logger` from `@utils/logger.js` — log every navigation with INFO level
- 404 handler: log a warning + redirect to `/`
- `route()` should throw a `Logger.error` (not throw) if the same path is registered twice
- Do not couple to any view — handlers are passed in from `routes.js`

---

### Task 7.2 — `src/js/config/routes.js`

**What it is:** Central registry of all app routes. Maps URL patterns to view functions.

**Subtasks:**

- [ ] Create `src/js/config/routes.js`
- [ ] Import `route` from `router.js`
- [ ] Use **dynamic imports** (`import()`) for all views — not static imports — so views load only when needed
- [ ] Register these routes: `/` → home, `/university/:id`, `/faculty/:id`, `/department/:id`, `/subject/:id`
- [ ] Register `/admin` route ONLY if `ENV.isDev === true`
- [ ] Export `registerRoutes()` function

**Coding rules:**

- **Dynamic imports are required** — static imports would crash if view files don't exist yet
  ```javascript
  // ✅ Correct
  route('/', async () => {
    const { renderHomeView } = await import('../pages/home-view.js');
    renderHomeView();
  });
  // ❌ Wrong
  import { renderHomeView } from '../pages/home-view.js';
  ```
- Admin route must be dev-only — guarded by `ENV.isDev` — not just hidden, not registered at all

## Phase 8 — Services + Cache Layer

**What this phase does:** All Firestore data access. Views never call Firestore directly — they call services. Services use cache to minimize reads.

**Dependencies:** Phase 7 complete, `firebase.js` ready ✅

---

### Task 8.1 — `src/js/utils/cache.js`

**What it is:** Memory cache with TTL. Wraps any async function and caches the result.

**Subtasks:**

- [ ] Create `src/js/utils/cache.js`
- [ ] Internal store: `Map<key, { data, expiresAt }>`
- [ ] `withCache(key, ttlMs, fetchFn)` — if valid cache entry exists, return it; otherwise call `fetchFn()`, store result, return it
- [ ] `clearCache(key?)` — clears one key (or all if no key passed)
- [ ] `isCached(key)` — returns boolean
- [ ] Cache is in-memory only — resets on page reload (by design)

**TTL values to use in services:**
| Data | TTL |
|---|---|
| Universities | 30 minutes |
| Faculties | 30 minutes |
| Departments | 30 minutes |
| Subjects | 15 minutes |
| Questions (V0.5) | 5 minutes |
| Leaderboard (V0.5) | 2 minutes |
| User data | Never cache |

**Coding rules:**

- Cache keys must be descriptive and namespaced: `universities:all`, `faculties:by:cu-2024`
- When admin panel edits data — it must call `clearCache('universities:all')` after the write
- Do not cache user-specific sensitive data
