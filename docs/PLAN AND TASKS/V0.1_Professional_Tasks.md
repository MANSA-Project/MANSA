### **المهمة 1.3: استراتيجية الفروع (Git Flow)**

**ما هي استراتيجية الفروع؟**
نظام منظم لإدارة التطوير بحيث لا يؤثر عمل مطور على آخر.

**الاستراتيجية المطلوبة: Git Flow المبسط**

#### **الفروع الدائمة:**

**1. `main` (Production)**

- **الغرض:** الكود المستقر الجاهز للنشر
- **القواعد:**
  - لا push مباشر НИКОГДА
  - فقط عبر Pull Request من `develop`
  - يجب مراجعة كود (Code Review)
  - يجب نجاح جميع الاختبارات
- **متى يُحدّث:** عند إطلاق نسخة جديدة رسمياً
- **الحماية:** تفعيل Branch Protection Rules

**2. `develop` (Staging)**

- **الغرض:** التطوير النشط
- **القواعد:**
  - لا push مباشر
  - فقط عبر PR من feature branches
- **متى يُحدّث:** عند إكمال ميزة وتجربتها

#### **الفروع المؤقتة:**

**3. `feature/*` (ميزات جديدة)**

- **التسمية:** `feature/user-authentication`, `feature/question-bank`
- **ينشأ من:** `develop`
- **يُدمج في:** `develop`
- **العمر:** قصير (1-5 أيام)
- **يُحذف:** بعد الدمج

**4. `bugfix/*` (إصلاح أخطاء)**

- **التسمية:** `bugfix/login-validation`, `bugfix/navbar-mobile`
- **ينشأ من:** `develop`
- **يُدمج في:** `develop`

**5. `hotfix/*` (إصلاحات عاجلة)**

- **التسمية:** `hotfix/security-patch`, `hotfix/firebase-crash`
- **ينشأ من:** `main`
- **يُدمج في:** `main` **و** `develop`
- **متى:** خطأ خطير في Production

**6. `release/*` (تجهيز نسخة)**

- **التسمية:** `release/v0.1`, `release/v0.5`
- **ينشأ من:** `develop`
- **يُدمج في:** `main` و `develop`

#### **سير العمل اليومي:**

```
1. بدء ميزة جديدة:
   git checkout develop
   git pull origin develop
   git checkout -b feature/my-new-feature

2. العمل على الميزة:
   (تعديلات، commits)

3. الانتهاء:
   git push origin feature/my-new-feature
   (إنشاء Pull Request على GitHub)

4. بعد المراجعة والموافقة:
   (دمج في develop)
   git checkout develop
   git pull origin develop
   git branch -d feature/my-new-feature
```

**المهام الفرعية:**

- [ ] إنشاء فرع `develop` من `main`
- [ ] تفعيل Branch Protection على `main`:
  - Require pull request reviews (1+)
  - Require status checks to pass
  - Include administrators
- [ ] توثيق الاستراتيجية في README
- [ ] تدريب الفريق على السير

### **المهمة 1.4: اصطلاحات Commits (Conventional Commits)**

**ما هي Conventional Commits؟**
صيغة موحدة لرسائل الـ commits تجعل التاريخ واضحاً وقابلاً للفهم والأتمتة.

**الصيغة القياسية:**

```
<type>(<scope>): <subject>

<body>

<footer>
```

#### **الأنواع (Types) - إلزامية:**

| Type       | الاستخدام                       | مثال                                           |
| ---------- | ------------------------------- | ---------------------------------------------- |
| `feat`     | ميزة جديدة                      | `feat(auth): add email verification`           |
| `fix`      | إصلاح خطأ                       | `fix(navbar): resolve mobile menu not closing` |
| `docs`     | تعديل توثيق                     | `docs(readme): update installation steps`      |
| `style`    | تنسيق كود (لا يؤثر المعنى)      | `style(css): fix indentation in buttons.css`   |
| `refactor` | إعادة هيكلة (لا ميزة ولا إصلاح) | `refactor(utils): simplify date formatting`    |
| `perf`     | تحسين أداء                      | `perf(images): lazy load subject icons`        |
| `test`     | إضافة اختبارات                  | `test(auth): add login validation tests`       |
| `chore`    | مهام صيانة                      | `chore(deps): update firebase to v10.7`        |
| `ci`       | تعديل CI/CD                     | `ci(azure): add deployment workflow`           |
| `build`    | تعديل build system              | `build(vite): configure build output`          |
| `revert`   | التراجع عن commit               | `revert: feat(auth): add email verification`   |

#### **النطاق (Scope) - اختياري:**

مكون أو صفحة محددة تأثر بالتغيير.

**أمثلة:** `auth`, `navbar`, `firebase`, `css`, `i18n`, `api`

#### **الموضوع (Subject) - إلزامي:**

- جملة قصيرة بصيغة الأمر الحاضر
- بدون نقطة في النهاية
- حد أقصى: 50 حرف

**أمثلة جيدة:**

- ✅ `add user profile modal`
- ✅ `fix infinite loop in quiz timer`
- ✅ `update API endpoint for questions`

**أمثلة سيئة:**

- ❌ `Added user profile modal.` (ماضي + نقطة)
- ❌ `fix` (غير واضح)
- ❌ `updates and improvements` (عام جداً)

#### **الجسم (Body) - اختياري:**

شرح مفصل للتغيير - **لماذا** وليس **ماذا**.

```
fix(auth): prevent duplicate email registration

Previously, users could register twice with the same email
due to missing uniqueness constraint. This caused data
inconsistency in Firestore.

This commit adds email validation before registration.
```

#### **التذييل (Footer) - اختياري:**

معلومات إضافية: Breaking Changes, Issues.

```
feat(api): migrate to REST API v2

BREAKING CHANGE: API endpoint changed from /v1 to /v2
Closes #123, #124
```

#### **Breaking Changes:**

عندما يكسر الكود التوافق مع الإصدار السابق:

```
feat(auth)!: change login API signature

BREAKING CHANGE: login() now requires email and password as object
```

**المهام الفرعية:**

- [ ] إنشاء ملف `CONTRIBUTING.md` مع قواعد الـ commits
- [ ] إنشاء قالب commit message في `.gitmessage`:

```
# <type>(<scope>): <subject>
# |<----  max 50 chars  ---->|

# Body - explain WHY (optional)
# |<----  max 72 chars  ---->|

# Footer (optional)
# - Breaking changes
# - Issue references

# Types: feat, fix, docs, style, refactor, perf, test, chore
```

- [ ] تفعيل القالب: `git config commit.template .gitmessage`
- [ ] مشاركة مع الفريق

### **المهمة 2.3: إنشاء ملفات Environment**

**الغرض:**
فصل الإعدادات حسب البيئة (تطوير/إنتاج).

**ما هي ملفات .env؟**
ملفات تحتوي متغيرات بيئية (API keys, database URLs) التي تختلف بين التطوير والإنتاج.

**المهام الفرعية:**

**1. إنشاء `.env.example` (يُرفع لـ Git):**

```bash
# Firebase Configuration
VITE_FIREBASE_API_KEY=
VITE_FIREBASE_AUTH_DOMAIN=
VITE_FIREBASE_PROJECT_ID=
VITE_FIREBASE_STORAGE_BUCKET=
VITE_FIREBASE_MESSAGING_SENDER_ID=
VITE_FIREBASE_APP_ID=

# API Keys
VITE_GROQ_API_KEY=
VITE_GEMINI_API_KEY=

# Environment
NODE_ENV=development
```

**لماذا `VITE_` prefix؟**
Vite يعرض فقط المتغيرات التي تبدأ بـ `VITE_` للمتصفح (أمان).

**2. إنشاء `.env.development` (لا يُرفع):**

```bash
VITE_FIREBASE_API_KEY=your_dev_api_key
# ... باقي المفاتيح التطويرية
NODE_ENV=development
```

**3. إنشاء `.env.production` (لا يُرفع):**

```bash
VITE_FIREBASE_API_KEY=your_prod_api_key
# ... باقي المفاتيح الإنتاجية
NODE_ENV=production
```

**4. إنشاء `src/js/config/env.js`:**

```javascript
/**
 * Environment Configuration
 * Loads variables from .env files via Vite
 */

export const ENV = {
  // Environment
  NODE_ENV: import.meta.env.MODE,
  isDev: import.meta.env.DEV,
  isProd: import.meta.env.PROD,

  // Firebase
  firebase: {
    apiKey: import.meta.env.VITE_FIREBASE_API_KEY,
    authDomain: import.meta.env.VITE_FIREBASE_AUTH_DOMAIN,
    projectId: import.meta.env.VITE_FIREBASE_PROJECT_ID,
    storageBucket: import.meta.env.VITE_FIREBASE_STORAGE_BUCKET,
    messagingSenderId: import.meta.env.VITE_FIREBASE_MESSAGING_SENDER_ID,
    appId: import.meta.env.VITE_FIREBASE_APP_ID,
  },

  // AI APIs
  groqApiKey: import.meta.env.VITE_GROQ_API_KEY,
  geminiApiKey: import.meta.env.VITE_GEMINI_API_KEY,
};

// Validation
if (ENV.isProd && !ENV.firebase.apiKey) {
  throw new Error('Missing Firebase API Key in production');
}
```

**كيف يعمل؟**

1. Vite يقرأ `.env.development` في التطوير
2. Vite يقرأ `.env.production` عند البناء
3. `import.meta.env` يوفر الوصول للمتغيرات

```
[overlay - خلفية معتمة]

       │     └── [close button]
```

## 🌐 المرحلة 5: نظام الترجمة (i18n)

### **المهمة 5.1: بناء وحدة i18n**

**ما هو i18n؟**
اختصار لـ internationalization - دعم لغات متعددة.

**لماذا من اليوم الأول؟**
إضافته لاحقاً يعني تعديل كل نص في التطبيق (مئات الأماكن). بناؤه الآن = توفير أيام عمل.

**كيف يعمل النظام؟**

```
HTML                     i18n.js              ar.js
─────────────────────    ─────────────────    ─────────────────
<h1 data-i18n="title">   ──► t('title')  ──►  { title: "ليالي الامتحان" }
<button data-i18n="btn"> ──► t('btn')    ──►  { btn: "إرسال" }
```

**عند تغيير اللغة:**

1. تُحمّل `en.js`
2. يُغيّر `dir` من `rtl` إلى `ltr`
3. يُحدّث كل عنصر فيه `data-i18n`

**المهام الفرعية:**

- [ ] إنشاء `src/js/config/i18n.js`

**الدوال الأساسية:**

**1. `loadTranslations(lang)`:**

- تحمّل ملف الترجمة ديناميكياً (`ar.js` أو `en.js`)
- ترجع `Promise` ينتهي بعد تحميل الملف
- تستخدم `dynamic import()` وليس `<script>` tag

**لماذا dynamic import؟**
يحمّل الملف فقط عند الحاجة (lazy loading). لا نحمّل ملف الإنجليزية إذا اللغة عربية.

```javascript
const translations = await import(`./translations/${lang}.js`);
```

**2. `t(key, vars = {})`:**

- تُرجع النص المترجم للـ key
- تدعم Template Variables:
  - Key: `"welcome"` → Value: `"مرحباً {name}!"`
  - استخدام: `t('welcome', { name: 'أحمد' })` → `"مرحباً أحمد!"`
- إذا لم يُجد الـ key: يرجع الـ key نفسه (لتسهيل debugging)

  **3. `setLanguage(lang)`:**

- تحفظ اللغة في localStorage
- تستدعي `loadTranslations(lang)`
- تغيير `document.documentElement.lang`
- تغيير `document.documentElement.dir` (rtl/ltr)
- تستدعي `applyTranslations()`

**4. `applyTranslations()`:**

- تمر على كل عنصر فيه `data-i18n`
- تستبدل النص بـ `t(key)`
- تمر على كل عنصر فيه `data-i18n-placeholder`
- تحدّث الـ placeholder

```javascript
document.querySelectorAll('[data-i18n]').forEach(el => {
  const key = el.dataset.i18n;
  el.textContent = t(key);
});
```

**5. `getCurrentLang()`:**

- تُرجع اللغة الحالية من localStorage
- افتراضي: `'ar'`

**6. `initI18n()`:**

- تُستدعى مرة واحدة عند بدء التطبيق
- تقرأ اللغة المحفوظة
- تحمّل الترجمة
- تطبّق على الصفحة

### **المهمة 6.3: Service Worker**

**ما هو Service Worker؟**
سكريبت JavaScript يعمل في background thread منفصل عن الصفحة.
يتحكم في الطلبات بين المتصفح والشبكة.

**لماذا لا يمكنه الوصول للـ DOM؟**
لأنه يعمل في thread منفصل - هذا يجعله سريعاً ولا يعيق الـ UI.

**الاستراتيجيات الثلاثة الرئيسية:**

**1. Cache First (للأصول الثابتة - CSS, JS, Images):**

```
طلب ملف → هل في Cache؟
    نعم → استخدم Cache (سريع جداً)
    لا → جلب من Network → حفظ في Cache → رجوع
```

**2. Network First (للـ API calls والبيانات):**

```
طلب بيانات → جرّب Network أولاً
    نجح → رجوع + حفظ في Cache
    فشل → استخدم Cache القديم (offline fallback)
```

**3. Stale While Revalidate (للمحتوى الديناميكي):**

```
طلب صفحة → أرجع Cache فوراً (سريع)
           + جلب من Network في الخلفية
           + حدّث Cache للمرة القادمة
```

**المهام الفرعية:**

- [ ] إنشاء `public/service-worker.js`

**البنية الأساسية:**

**1. التهيئة:**

```javascript
const CACHE_VERSION = 'v1';
const STATIC_CACHE = `static-${CACHE_VERSION}`;
const DYNAMIC_CACHE = `dynamic-${CACHE_VERSION}`;

// الملفات الأساسية للـ cache
const STATIC_ASSETS = [
  '/',
  '/index.html',
  '/manifest.json',
  '/src/css/main.css',
  '/src/js/main.js',
  // خطوط، أيقونات...
];
```

**2. Install Event:**

```javascript
self.addEventListener('install', event => {
  event.waitUntil(
    caches
      .open(STATIC_CACHE)
      .then(cache => cache.addAll(STATIC_ASSETS))
      .then(() => self.skipWaiting()) // تفعيل فوري
  );
});
```

**`self.skipWaiting()`:**
بدونه: Service Worker الجديد ينتظر حتى تُغلق كل التبويبات
معه: يتفعّل فوراً

**3. Activate Event:**

```javascript
self.addEventListener('activate', event => {
  event.waitUntil(
    caches
      .keys()
      .then(keys => Promise.all(keys.filter(key => key !== STATIC_CACHE && key !== DYNAMIC_CACHE).map(key => caches.delete(key))))
      .then(() => self.clients.claim()) // يسيطر على جميع التبويبات فوراً
  );
});
```

**4. Fetch Event (القلب):**

```javascript
self.addEventListener('fetch', event => {
  const { request } = event;
  const url = new URL(request.url);

  // API calls: Network First
  if (url.pathname.startsWith('/api/') || url.host.includes('firebase')) {
    event.respondWith(networkFirst(request));
    return;
  }

  // Static assets: Cache First
  if (request.destination === 'style' || request.destination === 'script' || request.destination === 'image') {
    event.respondWith(cacheFirst(request));
    return;
  }

  // HTML pages: Stale While Revalidate
  event.respondWith(staleWhileRevalidate(request));
});
```

### **المهمة 6.4: تسجيل Service Worker**

**المهام الفرعية:**

- [ ] في `src/js/main.js`:

```javascript
/**
 * Register Service Worker
 * Only in production and supported browsers
 */
async function registerServiceWorker() {
  if (!('serviceWorker' in navigator)) {
    console.log('Service Worker not supported');
    return;
  }

  try {
    const registration = await navigator.serviceWorker.register('/service-worker.js', {
      scope: '/',
    });

    // Check for updates every hour
    setInterval(() => registration.update(), 60 * 60 * 1000);

    registration.addEventListener('updatefound', () => {
      const newWorker = registration.installing;
      newWorker.addEventListener('statechange', () => {
        if (newWorker.state === 'installed' && navigator.serviceWorker.controller) {
          // New version available - show update notification
          Toast.info('تحديث جديد متاح! أعد تحميل الصفحة.');
        }
      });
    });
  } catch (error) {
    console.error('Service Worker registration failed:', error);
  }
}

// Register after page loads (لا يعيق تحميل الصفحة)
window.addEventListener('load', registerServiceWorker);
```

```yaml
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Build
        run: npm run build
        env:
          VITE_FIREBASE_API_KEY: ${{ secrets.VITE_FIREBASE_API_KEY }}
          VITE_FIREBASE_PROJECT_ID: ${{ secrets.VITE_FIREBASE_PROJECT_ID }}
          # ... باقي الـ secrets
```

**`npm ci` vs `npm install`:**

- `npm install` - يحدّث `package-lock.json`
- `npm ci` - يثبّت **بالضبط** ما في `package-lock.json` (أسرع وأكثر موثوقية في CI)

**كيف تُضيف Secrets؟**

2. New repository secret
3. أضف كل مفاتيح Firebase وAPI keys

```javascript
 *
```
